## ДЗ1:  

Тип: платформа коротких видео с алгоритмической рекомендательной лентой

Ключевой функционал(MVP):

1) Персонализированная лента видео
2) Загрузка видео
3) Лайки
4) Комментарии
5) Подписки
6) Поиск видео, авторов

Ключевые продуктовые решения:

1) Основной экран сервиса - рекомендательная лента, формируемая на основе взаимодействий пользователя (просмотры, досматриваемость, лайки, подписки).
   Рекомендации обновляются периодически с учётом новых действий пользователей

2) Загруженные видео сохраняются в распределённом объектном хранилище (origin storage) и доставляются пользователям через CDN, что снижает задержки воспроизведения и нагрузку на backend

3) После загрузки видео проходит асинхронную обработку: транскодирование в несколько качеств и генерацию превью. Видео становится доступным пользователям после завершения обработки

4) Видео доступны в нескольких вариантах качества, которое подбирается автоматически в зависимости от скорости сети и устройства пользователя

5) События просмотров, лайков и комментариев агрегируются перед записью в БД, поэтому счётчики могут обновляться с небольшой задержкой(eventual consistency)

6) Поиск реализуется по авторам, текстовым описаниям и хештегам видео с использованием полнотекстового индекса для быстрого поиска и ранжирования результатов.

#### Объём хранилища

**Расчёт:**
- Загрузок в день: 23 млн
- Хранимый размер одного видео: 70 MB
- Ежедневный прирост: `23 млн × 70 MB = 1 610 000 GB ≈ 1 572 ТБ/сутки`
- За год: `1 572 × 365 ≈ 573 780 ТБ ≈ 560 ПБ/год`

**Формулы:**
- Средний RPS = (DAU × действий/день) / 86 400 сек

|---|---:|---:|---:|---|

#### Сетевой трафик

**Суточный трафик:**
- Исходящий (CDN): `477 760 млн просмотров/день × 4.56 MB ≈ 2 178 272 ТБ/сутки ≈ **2 129 ПБ/сутки**`
- Входящий (upload): `23 млн загрузок/день × 20 MB ≈ 950 ТБ/сутки`

> Это теоретический максимум (верхняя оценка). Реальная цифра существенно ниже по нескольким причинам:
> - Не все пользователи досматривают ролик до конца
> - ABR (адаптивный битрейт) снижает фактический битрейт при плохом соединении
> - Часть трафика покрывается кешированием на CDN-edge без повторной передачи

> С поправочным коэффициентом ~0.5 реалистичная оценка: **~1 065 ПБ/сутки**

## ДЗ3: Глобальная балансировка нагрузки

### Функциональное разбиение по доменам

Сервис обслуживает три принципиально разных типа трафика: лёгкие API-запросы к backend, тяжёлый видеопоток через CDN и загрузку видео в origin storage. Домены выделены строго под функции MVP — по одному домену на каждый независимо масштабируемый слой.

| Домен | Назначение |
|---|---|
| `api.tiktok.com` | Backend API: лента, лайки, комментарии, подписки, поиск |
| `auth.tiktok.com` | Аутентификация, выдача и обновление токенов |
| `upload.tiktok.com` | Загрузка видео от пользователей в origin storage |
| `cdn.tiktok.com` | Стриминг видео и доставка превью через CDN-edge узлы |

---

### Расположение дата-центров

#### Ключевые факторы выбора

Аудитория сервиса географически неоднородна. Из топ-10 стран по числу пользователей складываются три крупных кластера примерно равного веса:

- **Юго-Восточная Азия** (Индонезия + Вьетнам + Филиппины + Таиланд): **~377 млн**
- **Латинская Америка** (Бразилия + Мексика): **~230 млн** — второй по размеру кластер
- **США**: **~153 млн**
- **Южная Азия** (Пакистан + Бангладеш): **~136 млн**
- **Россия**: **~67 млн**

Стриминг видео требует задержки **< 200 мс** до CDN-edge. Выбор ДЦ определяется двумя факторами:
1. **Близость к крупнейшим аудиториям** — ДЦ должны покрывать регионы с наибольшей концентрацией DAU
2. **Пересечение с подводными кабельными магистралями** — для минимизации межрегиональной задержки

## ДЗ4: Локальная балансировка нагрузки

### Схема балансировки нагрузки

Весь MVP-трафик (`api`, `auth`, `upload`) работает через HTTP/HTTPS. CDN-трафик (`cdn.tiktok.com`) проходит через CDN-провайдера и не нагружает наши балансировщики.
Используется один уровень балансировки — **L7 (NGINX)**.

NGINX выполняет:
- **SSL Termination** — расшифровка HTTPS на прокси, далее трафик идёт по HTTP внутри ДЦ
- **Keep-alive к backend** — постоянные соединения снижают overhead на установку TCP-соединений
- **Маршрутизация по доменам** — `api`, `auth`, `upload` направляются в соответствующие пулы backend-серверов
- **Least Connections** — алгоритм балансировки, равномерно распределяющий нагрузку с учётом текущей загруженности каждого backend

Резервирование: **N+1 (Active-Active)**. Все серверы работают одновременно. Health check периодически проверяет доступность каждого узла и исключает упавшие из ротации.
